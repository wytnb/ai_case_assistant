# 文档策略（docs-policy）

本文件定义项目文档体系的完整清单、适用条件、模板、示例、命名规则，以及新建文档的判定逻辑。  
`AGENTS.md` 负责要求 Codex 遵守；本文件负责告诉 Codex 和维护者具体应该怎么判断、怎么写、写到哪里。

当前目录属于 monorepo 下的服务文档：

- 根级 `docs/` 负责 workspace 与业务本体事实
- 根级 `contracts/` 负责共享机器可读契约
- 当前 `services/ai_gateway/docs/` 负责 Worker 实现事实

## 1. 目的

本仓库的文档体系需要同时满足以下目标：

1. 新接手的人读完文档后，可以直接启动项目并理解关键约束
2. Codex 执行任务时，不只更新代码，也会同步更新文档与测试
3. 代码任务结束后，Codex 能判断并补齐所需测试层，再执行本地、live 和线上验证
4. 项目在频繁需求变更下，仍能保持边界清晰、影响面可追踪、回归口径可落地
5. Codex 能根据 `AGENTS.md` + `docs/docs-policy.md` 自行判断是否需要新增文档

## 2. 总原则

### 2.1 优先更新已有文档

默认先更新已有文档，而不是新建文档。

### 2.2 新建文档必须有充分理由

只有当信息已经形成独立、长期、可复用主题时，才应新建文档。

### 2.3 一份文档只负责一个主题

避免把范围、接口、架构、测试策略混写在同一个文档里。

### 2.4 业务事实与 AI 规则分离

- `docs/` 记录项目事实
- `AGENTS.md` / `.cursor/rules/` 记录执行规则

### 2.5 文档必须支持变更管理

文档不能只描述“现在是什么”，还要支持回答：

- 为什么这样设计
- 改了什么
- 哪些地方会受影响
- 这次至少该补哪些测试
- 需要做哪些本地、live、线上验证

### 2.6 不编造事实

文档内容必须来自以下至少一种证据：

- 代码实现
- 配置文件
- 现有文档
- 测试
- 明确可推断的仓库结构事实

如果信息不足，写“待确认”，不要补写臆测规则。

## 3. 何时更新已有文档，何时新建文档

### 3.1 默认流程

在决定是否新建文档时，按以下顺序判断：

1. 这次改动能否清晰归入现有文档主题？
   - 能：更新已有文档
2. 这次改动是否只是一小段补充、修正、澄清？
   - 是：更新已有文档，不新建
3. 这次改动是否形成新的长期主题？
   - 否：不新建
4. 这次改动是否会在后续任务中被反复引用？
   - 否：通常不新建
5. 单独成文是否能显著降低未来维护成本？
   - 否：通常不新建
6. 以上都满足时：允许新建文档

### 3.2 不应新建文档的情况

以下情况通常不应新建文档：

- 一次性任务说明
- 小范围字段修正
- 局部 bug 修复，且已有文档可以承接
- 临时排查记录
- 给已有文档补一小节说明
- 某次任务中的过渡性思考，不具长期复用价值

### 3.3 应考虑新建文档的情况

以下情况应考虑新建文档：

- 新增长期存在的业务主题
- 新增独立子系统或独立模块
- 新增一类会被反复引用的规则集合
- 新增复杂流程，已有文档已无法清晰承载
- 重要架构决策需要单独记录取舍
- 现有文档继续膨胀会明显降低维护性

## 4. 文档总清单

### 4.1 给人看的文档

#### README.md

- 用途：项目入口，说明项目是什么、怎么跑、去哪看详细文档
- 适用条件：所有项目必备
- 更新时机：启动方式、关键目录、项目定位变化时
- 是否允许 Codex 自动创建：是

#### docs/00-index.md

- 用途：文档导航与简要用途说明
- 适用条件：所有项目必备
- 更新时机：文档新增、删除、重命名时
- 是否允许 Codex 自动创建：是

#### docs/01-overview.md

- 用途：项目概览、目标用户、核心问题、成功标准
- 适用条件：所有业务项目
- 更新时机：目标用户、项目目标、核心价值变化时
- 是否允许 Codex 自动创建：是

#### docs/02-scope-and-nongoals.md

- 用途：明确当前范围、边界、不做什么
- 适用条件：需求容易膨胀的项目
- 更新时机：范围调整、优先级变化、延期项变化时
- 是否允许 Codex 自动创建：是

#### docs/03-business-flows.md

- 用途：主流程、异常流程、状态流转
- 适用条件：存在业务流程的项目
- 更新时机：流程、状态、异常处理变化时
- 是否允许 Codex 自动创建：是

#### docs/04-domain-model.md

- 用途：核心对象、关系、字段语义、不变量
- 适用条件：存在核心业务实体时
- 更新时机：实体、字段、关系、不变量变化时
- 是否允许 Codex 自动创建：是

#### docs/05-system-architecture.md

- 用途：模块划分、依赖边界、技术路线
- 适用条件：所有中长期项目
- 更新时机：模块边界、架构路线、依赖关系变化时
- 是否允许 Codex 自动创建：是

#### docs/06-api-contracts.md

- 用途：接口契约、输入输出、错误码、兼容要求
- 适用条件：有内部或外部接口时
- 更新时机：接口或契约变化时
- 是否允许 Codex 自动创建：是
  - 备注：当前公开 HTTP source of truth 在根级 `../../contracts/health-record-ai.openapi.json`

#### docs/07-data-model.md

- 用途：数据库或存储结构、迁移、索引、兼容要求
- 适用条件：有持久化时；无持久化时用于明确记录“当前无”
- 更新时机：数据结构、绑定、迁移逻辑变化时
- 是否允许 Codex 自动创建：是

#### docs/08-rules-and-edge-cases.md

- 用途：规则表、优先级、边界条件、异常处理
- 适用条件：规则复杂、边界多的项目
- 更新时机：规则或异常逻辑变化时
- 是否允许 Codex 自动创建：是

#### docs/09-env-and-runbook.md

- 用途：环境说明、配置项、本地运行、live 测试、部署、排障
- 适用条件：多环境或外部依赖较多时
- 更新时机：环境、配置、部署流程变化时
- 是否允许 Codex 自动创建：是

#### docs/10-testing-strategy.md

- 用途：测试分层、覆盖要求、测试原则、验证口径
- 适用条件：所有中长期项目
- 更新时机：测试口径变化时
- 是否允许 Codex 自动创建：是

#### docs/11-regression-matrix.md

- 用途：变更影响面与回归矩阵
- 适用条件：模块较多、变更影响面较大时
- 更新时机：模块或高风险流程变化时
- 是否允许 Codex 自动创建：是

#### docs/12-release-smoke-checklist.md

- 用途：发布前后 smoke 检查
- 适用条件：存在部署、线上验证或发布行为时
- 更新时机：发布流程、关键验收点或 smoke 脚本变化时
- 是否允许 Codex 自动创建：是

#### docs/13-requirement-deltas.md

- 用途：记录需求变更、原因、影响面、补救动作
- 适用条件：需求频繁变化时强烈建议有
- 更新时机：需求含义、优先级、方案、关键假设发生变化时
- 是否允许 Codex 自动创建：是

#### docs/adr/ADR-xxxx-*.md

- 用途：记录重要架构决策与取舍
- 适用条件：发生架构、模块边界、兼容策略、技术路线决策时
- 更新时机：新增重要决策或旧决策被推翻时
- 是否允许 Codex 自动创建：是

### 4.2 给 AI 看的文档

#### AGENTS.md

- 用途：项目级执行规则、文档决策规则、测试与完成定义
- 适用条件：使用 Codex 时建议必备
- 更新时机：团队协作规则、完成定义、文档策略变化时
- 是否允许 Codex 自动创建：是，但后续应谨慎修改

#### .cursor/rules/00-core.mdc

- 用途：Cursor Agent 核心规则
- 适用条件：使用 Cursor 时建议必备
- 更新时机：核心项目规则变化时
- 是否允许 Codex 自动创建：是

#### .cursor/rules/10-docs-and-tests.mdc

- 用途：约束文档同步、测试同步与验证汇报
- 适用条件：希望 AI 稳定补文档和测试时
- 更新时机：文档或测试规则变化时
- 是否允许 Codex 自动创建：是

## 5. 变更类型 -> 应更新的文档

| 变更类型 | 至少应检查/更新的文档 |

| 主流程变化 | `03-business-flows.md`, `08-rules-and-edge-cases.md`, `11-regression-matrix.md` |

| 接口变化 | `06-api-contracts.md`, `10-testing-strategy.md`, `11-regression-matrix.md` |
| 数据结构或绑定变化 | `07-data-model.md`, `09-env-and-runbook.md`, `10-testing-strategy.md` |

## 7. 示例

### 示例 1：只是新增一个接口字段

情形：

- 在已有接口中新增一个可选字段
- 不改变主流程

通常应：

- 更新 `06-api-contracts.md`
- 如影响测试口径，更新 `10-testing-strategy.md`
- 更新 `11-regression-matrix.md`
- 增加相关测试

通常不应：

- 新建专题文档

### 示例 2：新增一个长期存在的新业务流程

情形：

- 新增一个后续会反复迭代的长期流程

通常应：

- 更新 `03-business-flows.md`
- 更新 `08-rules-and-edge-cases.md`
- 更新 `11-regression-matrix.md`

如果现有流程文档已过长、主题混杂，则可：

- 新建专题流程文档
- 并更新 `00-index.md`

### 示例 3：重构模块边界

情形：

- 拆分服务
- 调整核心模块职责
- 改变依赖方向

通常应：

- 更新 `05-system-architecture.md`
- 新增或更新 `ADR`
- 更新 `11-regression-matrix.md`

### 示例 4：修复一个已知 bug

情形：

- 修复某个边界条件下的错误行为

通常应：

- 更新相关主题文档
- 如规则口径变化，更新 `08-rules-and-edge-cases.md`
- 增加回归测试

通常不应：

- 新建新文档

## 8. 命名与放置规则

### 8.1 命名规则

- 文档统一使用小写英文文件名 + 连字符或数字前缀
- 保持稳定命名，避免频繁重命名

### 8.2 放置规则

- 通用项目文档放 `docs/`
- ADR 放 `docs/adr/`
- AI 规则文档放仓库根目录或 `.cursor/rules/`

### 8.3 索引规则

只要新增、删除、重命名文档，必须同步更新 `docs/00-index.md`。

## 9. 给 Codex / Cursor 的执行提示

当需求模糊时，应优先：

1. 补齐 `01-overview.md`
2. 补齐 `02-scope-and-nongoals.md`
3. 补齐 `03-business-flows.md`
4. 明确 `10-testing-strategy.md` 与 `11-regression-matrix.md` 中本次改动对应的测试要求
5. 再开始任务拆分与代码修改

不应直接跳过文档阶段去大规模写代码。

## 10. 仓库特定说明

- 当前测试目录是 `test/`，不是 `tests/`
- 当前仓库没有数据库、对象存储、队列或 CI/CD 配置
- 文档可以记录“当前无/未配置”，但不要为了对齐模板而伪造不存在的系统
- 默认把验证分成三层：
  - 本地测试：`npm test`
  - live 测试：`npm run test:live`
  - 线上 smoke：部署后按 `docs/12-release-smoke-checklist.md` 执行
- 如果本次任务应跑 live 或线上验证但无法执行，必须在最终汇报中明确阻塞原因
