# ADR-0001-单 Worker 同步调用 DeepSeek，并在本地做结构化校验与格式化

> 历史说明：本文关于公开端点范围的描述已被 [ADR-0003](./ADR-0003-retire-public-extract-endpoint.md) 部分覆盖；当前公开端点以 `docs/06-api-contracts.md` 为准。

## 背景

当前仓库实现的是一个最小化 Cloudflare Worker 服务：

- 核心逻辑集中在 `src/index.ts`
- 对外只暴露 `/ai/extract` 与 `/ai/report`
- 非空提取和报告流程都同步调用 DeepSeek
- 当前仓库没有数据库、队列、缓存或其他内部服务

与旧版本相比，当前提取链路已经不再在 Worker 内维护完整的相对时间推理规则，而是改为：

- 调用方显式提供 `eventTime`
- Worker 把 `eventTime` 作为唯一时间锚点传给 DeepSeek
- DeepSeek 返回结构化 `symptoms`
- Worker 本地做强校验、时间归一化和最终摘要格式化

## 决策问题

当前系统是否继续采用“单个无状态 Worker + 同步上游调用 + 本地结构化校验与格式化”的架构方式。

## 备选方案

1. 继续保持单个 Worker，同步调用 DeepSeek，并在 Worker 内做输入校验、上游校验与结果格式化
2. 拆分为多个模块或服务，将提取、报告和格式化逻辑分离
3. 在 Worker 内重新引入完整的本地时间推理引擎
4. 改为异步任务或队列模式处理 AI 请求

## 取舍分析

### 方案 1

优点：

- 部署形态简单
- 当前仓库已经按此实现
- 公开契约集中，便于维护
- `eventTime` 锚点可以由调用方控制，提取结果更可复现

缺点：

- 所有逻辑仍集中在单文件，后续膨胀时可维护性会下降
- 相对时间推断质量仍受模型行为影响

### 方案 2

优点：

- 职责边界更清晰
- 更利于长期扩展

缺点：

- 超出当前最小化服务形态
- 会增加重构成本和协作复杂度

### 方案 3

优点：

- 相对时间推断可完全由本地代码控制

缺点：

- 规则实现复杂度高
- 需要持续维护大量语言与边界规则
- 与当前“上游结构化结果 + 本地格式化”的方向相冲突

### 方案 4

优点：

- 更适合高延迟或高吞吐场景

缺点：

- 当前仓库没有队列和后台处理基础设施
- 会显著扩大系统边界

## 最终决策

当前继续采用方案 1：

- 单个无状态 Cloudflare Worker
- 同步调用 DeepSeek `chat/completions`
- `/ai/extract` 使用请求中的 `eventTime` 作为唯一时间锚点
- DeepSeek 返回结构化 `symptoms` 与 `notes`
- Worker 负责严格输入校验、上游 payload 校验、时间归一化与 `symptomSummary` 格式化

## 影响

- `/ai/extract` 对外契约更简单，只返回 `symptomSummary` 与 `notes`
- `/ai/report` 统一使用 `events[].eventTime`
- 本地不再维护旧的双时间字段推理逻辑
- 文档、测试和 smoke 需要持续围绕 `eventTime` 锚点维护

## 何时需要重新评估

出现以下任一情况时，应重新评估本 ADR：

- 提取时间推断的模型不稳定性成为主要问题
- `src/index.ts` 持续膨胀，影响维护效率
- 引入存储、缓存、队列或更多公开端点
- 需要支持远超当前复杂度的多区间症状时间建模
