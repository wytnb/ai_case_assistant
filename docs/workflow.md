# AI 工作流程规范

## 文档目的

本文件用于约束 AI 在本项目中接到任务后的工作流程、边界控制与输出格式。  
本文件不定义产品规则，不定义字段业务口径。

## AI 接到任务后的阅读顺序

### 默认顺序

1. `AGENTS.md`
2. 本文件
3. 与任务最相关的专用文档
4. 相关代码目录
5. 若任务涉及代码改动，在结束前必须阅读 `docs/doc_sync_matrix.md`

### 任务映射

- 页面任务：`product_facts` → `product_notes` → `architecture` → `contracts` → `acceptance`
- 数据模型任务：`architecture` → `contracts` → `acceptance`
- 交互优化任务：`product_notes` → `acceptance` → `conventions`
- Bug 修复任务：`architecture` → `contracts` → `acceptance`
- 文档同步任务：`doc_sync_matrix` → `acceptance`

## 改动前必须先确认的内容

1. 任务目标是什么
2. 本次只改什么
3. 本次明确不改什么
4. 涉及哪些页面、模块、数据对象
5. 是否涉及字段、状态、路由、存储、AI JSON
6. 是否会触碰架构边界
7. 是否需要数据库迁移或契约变更

若以上任一项不清楚，先收敛影响范围，不得直接大改。

## 改动前是否要先说明影响范围

默认要说明。  
说明至少包含：

- 改动模块
- 预计涉及文件类型
- 是否改动数据模型
- 是否改动路由
- 是否改动 AI 契约
- 是否存在回归点

对于非常小的局部修复，可以简化，但仍要体现“只动哪里”。

## 多模块任务怎么拆

### 原则

1. 先拆数据契约，再拆业务逻辑，再拆页面展示。
2. 先做能阻塞后续工作的基础部分。
3. 一个子任务只处理一个明确责任。

### 推荐拆法

- 第一步：确认数据对象与契约
- 第二步：确认仓储、数据源、用例
- 第三步：实现页面状态和页面骨架
- 第四步：补异常、空状态、边界交互
- 第五步：执行文档同步检查
- 第六步：自检验收点

## 不确定时如何收敛

1. 优先采用 MVP 最小实现。
2. 优先采用既有模式，不自创新模式。
3. 优先让数据可落库、可回显、可追溯。
4. 不确定是否应大改时，只给方案，不直接实施。
5. 不确定字段含义时，以 `docs/contracts.md` 为准；没有定义则先补契约，再写实现。

## 允许的顺手修改范围

在不扩大任务边界的前提下，只允许顺手做以下事项：

- 修正本次触达文件中的明显命名错误
- 补充缺失的基础空状态 / 错误处理
- 修正与本次功能直接相关的 import、类型、注释
- 删除本次触达文件中的明显死代码
- 同步更新受影响文档

## 禁止的顺手修改行为

- 顺手改目录结构
- 顺手统一全项目命名
- 顺手替换全局状态方案
- 顺手重构其他 feature
- 顺手修改无关页面 UI
- 顺手升级或替换依赖库
- 顺手扩展产品范围
- 顺手新增抽象层
- 跳过文档同步检查

## 何时只给方案，何时可以直接改代码

### 只给方案

以下情况默认只给方案，不直接改：

- 涉及数据库 schema 调整但影响面不清楚
- 涉及核心字段重命名
- 涉及路由重构
- 涉及跨多个 feature 的结构调整
- 涉及产品规则尚未明确
- 涉及破坏性迁移

### 可以直接改代码

以下情况可以直接改：

- 已明确的局部页面实现
- 已明确字段口径下的模型补全
- 已明确契约下的数据源或仓储实现
- 已明确页面的空状态、错误状态补全
- 已明确的 Bug 修复

## 输出结果必须包含什么

### 输出方案时

- 任务理解
- 改动范围
- 关键决策
- 默认假设
- 不处理项
- 风险点

### 输出代码或实现说明时

- 改了哪些模块
- 改了哪些文件类型
- 是否涉及数据模型 / 路由 / 契约
- 回归测试点
- 剩余风险或后续建议
- 文档同步结果

## 如何控制改动范围

1. 每次只围绕一个明确目标完成。
2. 不为“以后可能会用到”提前建大量抽象。
3. 除非用户明确要求，否则不跨模块统一风格。
4. 发现结构问题时，先标记，不自动升级为重构任务。
5. 一次任务里，新增功能与重构不能同时成为主线。

## 如何处理发现的额外问题

### 与本任务强相关

可一起处理，但必须在输出中说明“顺带修复了什么”。

### 与本任务弱相关

不直接处理，只在输出末尾简短列出。

### 与本任务无关

忽略，不展开。

## 如何避免一次任务引入无关重构

- 新代码优先贴合当前结构
- 只在本次触达文件范围内补齐必要规范
- 不因局部实现不美观而升级为全局重构
- 不在没有显式要求时调整基础设施

## 强制文档同步步骤

### 结束前必做

对任何修改代码的任务，在结束前必须执行 `docs/doc_sync_matrix.md` 定义的文档同步步骤。

### 检查顺序

1. 是否改动了用户可见行为  
   - 若是，检查 `docs/product_notes.md`、`docs/acceptance.md`、`README.md`、`docs/project_overview.md`
2. 是否改动了数据模型、字段、枚举、JSON、解析规则  
   - 若是，检查 `docs/contracts.md`
3. 是否改动了架构边界、模块职责、依赖方向、目录组织或路由组织原则  
   - 若是，检查 `docs/architecture.md` 或 `docs/architecture_notes.md`
4. 是否改动了命名、错误处理、日志、通用实现方式  
   - 若是，检查 `docs/conventions.md`
5. 是否改动了任务完成口径  
   - 若是，检查 `docs/acceptance.md`
6. 是否改动了 AI 协作方式  
   - 若是，检查 `AGENTS.md`、本文件、`docs/doc_sync_matrix.md`

### 结果要求

最终输出必须明确写出：

- 已更新的文档
- 未更新的文档
- 未更新原因

若无需更新任何文档，必须明确写出：

`No document update required.`

并附上理由。
