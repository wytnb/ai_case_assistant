# ADR-0002-MVP 阶段将业务编排保留在 Provider / 轻服务层

## 背景

- `health_record` 与 `report` 模块当前没有完整的 repository / use case / entity 分层。
- 业务编排主要放在 `presentation/providers/*.dart` 中的轻服务类和 Controller 里。
- `features/ai` 相对更清晰地拆出了 domain / data / provider。
- 页面层当前保持不直接调用 Dio、Drift 或文件系统。

## 决策问题

- 在当前 MVP 阶段，健康记录和报告模块是否要立即铺满完整分层抽象。

## 备选方案

### 方案 A：保持 Provider / 轻服务层编排

- 描述：页面通过 Provider 调用轻服务类，轻服务类直接协调数据库、AI 服务和本地文件存储。
- 优点：
  - 控制样板代码数量
  - 仍能保持页面层足够薄
  - 适合当前用例数量较少的阶段
- 缺点：
  - Provider 文件容易继续变大
  - 复杂度上升后拆分成本会增加

### 方案 B：立即补齐 repository / use case / entity 分层

- 描述：为 `health_record` 与 `report` 都建立完整领域与数据访问抽象。
- 优点：
  - 结构更规整，后续扩展边界更清晰
  - 更利于长期大型项目治理
- 缺点：
  - 当前阶段会引入较多样板代码
  - 对现有 MVP 用例来说抽象成本偏高

## 取舍分析

- 当前代码量和用例规模仍较小，最重要的是把新增记录、详情回显和报告生成闭环做稳。
- 页面层已经保持了“不过界”，因此立即铺满全部抽象层收益有限。
- 未来如果记录、报告、同步或复杂规则显著增多，再拆分更细的层次更合适。

## 最终决策

- 采用方案 A：MVP 阶段继续使用 Provider / 轻服务层承接业务编排。

## 影响

- 对代码结构的影响：
  - `health_record_providers.dart` 与 `report_providers.dart` 继续承接当前编排逻辑
  - 页面层保持只订阅状态与触发动作
  - 需要在文档中明确这是一种阶段性选择，而不是完整分层已经存在
- 对测试的影响：
  - 重点测试服务层逻辑、数据库交互和 Provider 驱动的状态变更
  - 当文件复杂度升高时，需要同步补更多模块级测试
- 对运维 / 发布的影响：
  - 对发布流程影响较小
  - 对维护者的主要要求是理解 Provider 文件中的编排逻辑

## 何时需要重新评估

- 单个 Provider 文件同时承载多个页面或多个数据源时
- 引入同步、编辑删除、更多 AI 流程后，业务规则明显增长时
