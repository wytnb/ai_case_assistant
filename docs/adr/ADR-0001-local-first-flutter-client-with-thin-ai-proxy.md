# ADR-0001-本地优先 Flutter 客户端与薄 AI 代理

## 背景

- 当前仓库使用 Flutter 客户端承载页面、路由、状态管理和本地持久化。
- 健康记录、附件与报告都保存在本地 Drift / 文件系统中。
- 当前公开 AI 能力通过 `/ai/intake` 与 `/ai/report` 两个 HTTP 接口接入。
- 仓库中没有账号体系、云同步、服务端业务编排或后台管理系统。

## 决策问题

- 在当前 MVP 阶段，系统应当以哪一层为主数据与业务编排中心。

## 备选方案

### 方案 A：本地优先 Flutter 客户端 + 薄 AI 代理

- 描述：客户端负责路由、状态、本地落库和附件存储，AI 代理只提供整理与报告接口。
- 优点：
  - 能快速形成可演示闭环
  - 离线时仍可查看已有本地数据
  - 降低服务端建设成本
- 缺点：
  - 数据仅保存在本地，缺少多端同步和备份
  - 客户端需要承担更多持久化和异常处理责任

### 方案 B：云端为中心的服务端业务系统

- 描述：客户端主要负责界面，记录、附件、报告和流程编排都交给后端。
- 优点：
  - 易于多端同步和统一管理
  - 服务端可集中实现权限、审计和后台管理
- 缺点：
  - 明显超出当前 MVP 的实现范围
  - 对部署、鉴权、运维和数据治理要求更高

## 取舍分析

- 当前仓库的代码、文档和测试都已围绕本地数据库、应用私有目录和薄 AI 接口展开。
- 项目目标是先完成“真实可演示 MVP”，而不是先建设完整云端体系。
- 因此当前不选择云端为中心方案。

## 最终决策

- 采用方案 A：本地优先 Flutter 客户端 + 薄 AI 代理。

## 影响

- 对代码结构的影响：
  - `core/database` 与附件本地存储成为主数据承载层
  - `features/intake` 承载新增记录主链路
  - `features/ai` 当前只承载报告接口与共享 AI 远程能力
  - 页面不直接处理原始 HTTP 响应
- 对测试的影响：
  - 需要重点覆盖数据库写入、迁移、AI 契约和本地回显
  - 真实 AI 集成测试保持为可选手工验证
- 对运维 / 发布的影响：
  - 当前主要维护客户端运行环境与 AI 代理地址
  - 暂不需要账号、同步、后台管理相关运维能力

## 何时需要重新评估

- 引入账号体系、云同步或多端共享时
- 报告与记录需要依赖服务端统一权限、审计或长期存储时
