# ADR-0003：客户端持有 intake session，未完成草稿与正式记录分离

## 状态

Accepted

## 日期

2026-03-19

## 背景

项目需要支持 AI 追问、继续补充、提前结束、离开后恢复，以及基于已完成正式记录再次补充并重新追问。

同时，当前 MVP 仍有以下边界：

- 不引入账号体系
- 不做云同步
- 不做服务端会话存储
- 不引入复杂医学结构化 schema

如果把追问会话托管给 worker 或把未完成草稿直接写入正式 `health_events`，会带来以下问题：

- 客户端无法在离开页面后可靠恢复本地上下文
- 未完成草稿容易污染正式记录列表与报告
- 重新追问更新原记录的链路会变得不清晰

## 决策

1. 追问会话由客户端 Drift 持有，新增 `intake_sessions`、`intake_messages`、`intake_session_attachments`。
2. 新增记录默认调用 `POST /ai/intake`。
3. 未完成追问草稿只保存在 `intake_*` 表中，不写入 `health_events`。
4. 正式记录完成后才把结果写入 `health_events`，并把会话暂存附件转正为 `attachments`。
5. 所有通过 `/ai/intake` 创建的正式记录都保留并复用对应 intake session。
6. 重新追问时复用原 session，并更新原正式记录，而不是创建重复记录。

## 结果

### 正面影响

- 客户端可以独立恢复未完成追问，无需服务端会话存储。
- 正式记录与未完成草稿边界清晰，报告查询不会混入草稿。
- 强制结束与重新追问都能复用统一的本地状态机。
- 首页设置与历史会话快照分离，便于解释会话行为。

### 负面影响

- 客户端状态机复杂度提升。
- 本地数据库与附件暂存需要额外迁移与测试。
- 旧 `/ai/extract` 历史记录不具备 session 关联能力。

## 备选方案

### 方案 A：把追问会话放到 worker

未采用，原因：

- 需要服务端会话存储，与当前 MVP 边界冲突。
- 本地恢复能力依赖网络和服务端状态。

### 方案 B：未完成草稿直接写入 `health_events`

未采用，原因：

- 会污染正式记录列表与报告。
- 无法准确表达“未完成追问”和“正式完成”的状态差异。

## 后续影响

- 后续若引入云同步或账号体系，需要重新评估 session 所有权与同步策略。
- 若未来引入更复杂的结构化字段，应继续保持“未完成草稿”和“正式记录”分层。
