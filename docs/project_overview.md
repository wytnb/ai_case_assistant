# 项目全貌说明

## 项目背景

AI 健康病例助手是一个单人开发的移动端 APP 项目。
项目当前主要用于作品展示、面试呈现和实践 AI 辅助开发，而不是直接面向商业化上线。

项目之所以成立，源于一个很常见的问题：普通人的健康信息通常散落在聊天记录、相册、纸质检查单和零碎记忆中，时间一长就很难回顾，也不利于就医前整理。

## 项目目标

项目希望完成三件事：

1. 让用户方便地记录健康相关输入
2. 让系统把零碎输入整理成更容易回顾的记录
3. 让用户能够基于一段时间内的记录得到清晰的汇总结果

当前阶段更强调“先做成、先演示、先形成闭环”，而不是一次性做成完整产品。

## 用户问题与解决思路

### 用户问题

- 健康信息分散，难以持续整理
- 就医前往往记不清近期症状和变化
- 检查单和病历图片保存了，但后续难检索、难总结
- 单纯记笔记容易缺少结构，不方便回看

### 解决思路

- 允许用户通过文字和图片记录健康信息
- 使用 AI 提炼摘要和备注
- 把记录沉淀为可回看、可汇总的健康事件
- 生成周报 / 月报 / 季报等周期性内容

## 当前版本范围

### 当前核心范围

- 首页入口
- 健康记录新增、列表、详情
- 本地附件保存与回显
- AI 提取摘要和备注

### 当前非核心但仍在方向内

### 当前不在范围内

- 账号体系
- 云同步
- 医疗机构对接
- 在线问诊
- 自动诊断
- 后台管理系统

## 当前真实实现进度

项目目前已经完成以下工作：

- Flutter / Android 开发环境已完成
- FVM 已就位
- 正式应用入口已替换默认 Counter 模板
- 基础路由已建立并接入首页、记录、报告页面
- Drift 本地数据库已建立
- `HealthEvents`、`Attachments`、`Reports` 三张表已落地
- 文字记录“新增 -> AI 提取 -> 本地保存 -> 列表 -> 详情”闭环已打通
- 图片附件“选择 -> 复制到应用私有目录 -> 入库 -> 详情回显”闭环已打通
- 报告“切换类型 -> 生成 -> 落库 -> 列表 -> 详情”闭环已打通
- AI 提取和 AI 报告接口已接入真实 HTTP 调用
- AI 提取保留了可切换 mock 的调试方式
- 文档体系与文档同步检查脚本已建立

## 当前页面结构

### 首页

- 展示项目标题
- 提供记录列表、新增记录、报告页三个入口
- 当前仍是静态入口页

### 健康记录模块

- 列表页展示摘要、时间和来源
- 新增页支持文本输入、多图选择、提交保存
- 详情页展示原始文本、摘要、备注和附件
- 图片附件支持点击后全屏预览

### 报告模块

- 列表页支持切换周报 / 月报 / 季报
- 生成按钮会触发当前选中类型的报告生成
- 详情页展示摘要、建议和 Markdown 原文

## 当前架构概览

系统采用以客户端为中心的本地优先架构。

### 客户端职责

- 页面展示与交互
- 本地结构化数据保存
- 附件本地复制与读取
- 记录与报告展示

### AI 代理职责

- 承接 AI 提取请求
- 承接 AI 报告请求
- 返回当前客户端可解析的 JSON

### 本地数据职责

- 保存健康记录、附件和报告
- 作为页面展示的主数据源
- 在无网络时继续提供历史记录浏览能力

## 当前限制

- 首页还没有承载最近记录或最近报告概览
- `sourceType` 当前统一写成 `text`
- 图片不会参与 AI 提取
- 报告详情当前未做 Markdown 富渲染
- AI 提取与报告生成都依赖网络
- 当前没有 settings、followup、编辑删除等扩展页面
- 当前测试覆盖较少

## 为什么当前这样做

- 先打通真实可演示闭环，而不是继续停留在架构设想
- 先验证本地保存、AI 提取、报告生成这些最关键路径
- 先控制抽象层数量，避免在 MVP 早期就堆大量 repository / use case 样板代码

## 后续迭代方向

1. 让首页承载更多真实数据概览
2. 为记录和报告模块补更多测试
3. 继续收敛 Provider 与服务类边界
4. 引入 AI 追问和更细粒度结构化结果
5. 评估记录编辑 / 删除、隐私保护与数据导出
