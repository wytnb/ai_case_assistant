# 项目全貌说明

## 项目背景

AI 健康病例助手是一个单人开发的移动端 APP 项目。  
项目当前主要用于作品展示、面试呈现和实践 AI 辅助开发，而不是直接面向商业化上线。

项目之所以成立，源于一个很常见的问题：普通人的健康信息通常散落在聊天记录、相册、纸质检查单和零碎记忆中，时间一长就很难回顾，也不利于就医前整理。

## 项目目标

项目希望完成三件事：

1. 让用户方便地记录健康相关输入
2. 让系统把零碎输入整理为结构化健康事件
3. 让用户能够基于一段时间内的记录得到清晰的回顾结果

当前阶段更强调“先做成、先演示、先形成闭环”，而不是一次性做成完整产品。

## 用户问题与解决思路

### 用户问题

- 健康信息分散，难以持续整理
- 就医前往往记不清近期症状和变化
- 检查单和病历图片保存了，但后续难检索、难总结
- 单纯记笔记容易缺少结构，不方便回看

### 解决思路

- 允许用户通过文字、图片、语音记录健康信息
- 使用 AI 做简单追问和结构化提取
- 把记录沉淀为可回看、可汇总的健康事件
- 生成周报 / 月报 / 季报等周期性内容
- 后续可延展到就医前摘要

## 核心功能结构

### 1. 记录输入

用户可以通过文本、图片、语音等方式输入健康信息。  
输入的目标不是单纯保存原文，而是为后续结构化整理提供原材料。

### 2. AI 追问

当原始输入信息不足时，系统可提出少量补充问题，帮助完善信息上下文。

### 3. 结构化提取

系统将用户输入和追问结果整理成结构化健康事件，方便后续展示、查询和汇总。

### 4. 历史记录浏览

用户可以查看已保存的健康事件、附件和提取结果。

### 5. 周期性报告

系统基于某一时间范围内的健康事件生成汇总型报告。

## MVP 范围

当前 MVP 以 Android 可演示闭环为目标，优先包括：

- 文本输入
- 图片输入
- 本地保存
- 列表和详情展示
- AI 提取基本链路
- 报告列表和详情

MVP 的核心价值不是“功能多”，而是“从输入到输出链路完整”。

## 技术方案概览

- Flutter：客户端框架
- Riverpod：状态管理与依赖注入
- go_router：路由管理
- Dio：网络访问
- freezed / json_serializable：模型代码生成
- Drift：本地结构化存储
- 本地图片存储：保存原始附件
- WorkManager：周期任务与补算入口
- 极薄 AI API 代理：承接 AI 模型调用

## 架构概览

系统采用以客户端为中心的离线优先架构。

### 客户端职责

- 页面展示与交互
- 本地结构化数据保存
- 附件本地存储
- 历史记录浏览
- 报告展示

### AI 代理职责

- 隐藏模型密钥
- 转发 AI 请求
- 提供稳定 JSON 接口边界

### 本地数据职责

- 作为主数据源保存健康事件、附件、提取结果、报告
- 保证页面在无网络时仍能展示已有记录

## 数据流概览

### 记录链路

用户输入  
→ 页面收集  
→ 调用 AI 追问 / 提取  
→ 结构化结果落本地  
→ 列表与详情展示

### 报告链路

本地健康事件集合  
→ 触发报告生成  
→ 结果落本地  
→ 列表与详情展示

## 当前限制

- 当前主要优先 Android 端
- 当前默认单用户本地使用
- 当前不做云同步
- AI 输出需要解析兜底，不能直接假设稳定
- 后台任务能力目前只作为增强，不作为唯一依赖

## 后续迭代方向
