# 架构说明与阶段性取舍

## 文档目的

本文件记录当前架构选择背后的原因、已知限制、可接受的技术债和未来可能调整的点。  
本文件不替代架构事实文档。

## 为什么采用当前技术栈

### Flutter

当前目标是尽快做出 Android 真机可演示成品，同时保留未来跨平台潜力。  
Flutter 适合单代码仓快速交付移动端 MVP，并且 AI 代码生成支持较成熟。

### Riverpod

需要一个同时承担依赖注入、异步状态管理和页面状态组织的统一方案。  
Riverpod 比简单的 `setState` 更适合中期扩展，也避免过早引入过重框架。

### go_router

当前项目存在列表、详情、报告、设置等多页面场景，路由需要统一收口。  
go_router 能让路由声明更清晰，适合中小型 Flutter 项目。

### Dio

需要明确的网络客户端层，后续要调用极薄 AI 代理。  
Dio 的拦截器、超时、错误处理能力比直接使用裸 HTTP 更适合项目演进。

### freezed / json_serializable

当前项目涉及本地模型、远程 JSON、状态枚举和不可变对象。  
使用代码生成可以减少手写样板，降低 AI 输出风格漂移。

### Drift

当前数据以结构化健康记录为主，后续需要查询、排序、筛选、按时间聚合。  
Drift 比纯 key-value 或文档式存储更适合此类查询与迁移需求。

### WorkManager

报告生成存在周期性触发需求，但当前不希望引入远程调度系统。  
WorkManager 允许在本地做轻量级后台任务或补算入口。

## 为什么采用当前架构边界

### 客户端主导

当前产品是单人开发、面试展示导向的 MVP。  
如果一开始就把复杂逻辑放到服务端，会明显拉高开发与维护成本，不符合当前目标。

### 本地结构化数据优先

健康记录天然带有连续时间线和本地隐私属性。  
先把本地模型、查询和展示做好，比先做云端更符合 MVP 收益。

### 薄代理而不是直连模型 API

客户端直连模型 API 会暴露密钥，也不利于未来做限流、日志和模型替换。  
薄代理既满足专业性，也把复杂度控制在可接受范围。

### feature + layer 双维度组织

只按 layer 组织会在业务扩展后变得难找代码；只按 feature 组织又容易把分层边界打乱。  
当前方案兼顾可读性和边界清晰度。

## 当前已知限制

1. 在完全离线时，只能浏览本地已有数据；无法完成新的 AI 提取与报告生成。
2. 图片本地存储在 MVP 阶段不做跨设备同步。
3. WorkManager 在不同 Android 设备上的触发时机可能不完全一致，因此报告补算不能只依赖后台任务。
4. 语音链路可能先以“转写文本”形式接入，音频文件本身不一定在第一版长期保存。
5. 若早期页面直接依赖部分临时 Provider 组合，后续可能需要再收敛状态职责。

## 技术债说明

1. 第一版可能先用单一 AI 代理实现追问、提取、报告三个接口，后续再拆更细能力。
2. 某些 ViewModel 或页面状态对象可能在早期与业务对象边界不够理想，但只要不越层可暂时接受。
3. 为了快速演示，部分复杂筛选、分页、缓存失效策略可能先不做。
4. 若初期没有本地加密，属于已知风险，但在当前阶段可暂缓。

## 阶段性妥协说明

1. 报告可先由用户主动触发，不强依赖稳定后台定时任务。
2. 某些 feature 可以先只有一个页面入口，再随功能增长拆更细页面。
3. 若某些通用 UI 尚未稳定，可先在 feature 内保留私有组件，不急于抽到 shared。
4. 只要契约稳定，局部实现允许先采用较薄的 use case 封装。

## 后续可能的演进方向

1. 增加本地加密与隐私保护能力。
2. 评估轻量云同步或备份能力。
3. 按需要拆分 AI 模块中的远程调用、解析、结果标准化职责。
4. 当页面复杂度明显上升时，进一步细化页面状态对象与交互动作对象。
5. 当报告与摘要能力增多时，考虑将“分析输出”从 `report` 中拆出独立 feature。

## 未来可能调整的点

- 数据库表的部分辅助字段
- 报告触发策略
- 语音输入保存策略
- AI 代理部署平台
- 设置模块中的调试与环境配置方式
