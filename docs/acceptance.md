# 验收标准

## 文档目的

本文件定义功能、页面、交互、数据、异常处理和文档同步在 MVP 阶段的最低完成口径。  
本文件回答“做到什么程度才算完成”，不描述实现过程和字段细节。

## 通用验收原则

1. 功能必须可演示，不以“代码已写完”作为完成标准。
2. 页面必须覆盖加载、成功、空、失败四类基本状态中的适用部分。
3. 本地数据必须可写入、可读取、可回显。
4. AI 相关能力必须在成功与失败两种情况下都可被用户感知和理解。
5. 不要求完美，但不能以崩溃、无提示、无回显、无恢复路径结束流程。

## 页面级验收标准

## 首页

完成标准：

- 能看到健康记录入口
- 能看到最近记录或空状态
- 能进入新增记录流程
- 能进入报告列表或报告入口

不合格情形：

- 首页没有明确主入口
- 数据为空时页面只是一片空白
- 首页依赖 AI 网络成功才能正常打开

## 新增记录页

完成标准：

- 至少支持文本录入
- 若支持图片录入，则能看到附件已添加的反馈
- 可以提交形成健康事件
- 提交失败时有明确提示

不合格情形：

- 输入后无法判断是否保存成功
- 图片添加后没有任何可见反馈
- 页面离开后数据无任何保存结果

## AI 追问页

完成标准：

- 能展示问题列表或空问题状态
- 能填写或跳过回答
- 能继续进入提取流程或返回记录详情
- AI 请求失败时有重试或返回路径

不合格情形：

- 用户不知道当前是否还需要操作
- 失败后页面卡死在 loading
- 回答内容丢失且无任何提示

## 记录详情页

完成标准：

- 能看到原始输入的至少一种形式
- 能看到附件列表或无附件状态
- 能看到结构化结果或失败提示
- 能看到事件时间和当前状态

不合格情形：

- 详情页无法区分原始输入与结构化结果
- 失败后没有任何提示
- 用户无法判断这条记录是否处理完成

## 报告页

完成标准：

- 能看到报告列表或空状态
- 能打开报告详情
- 报告生成失败时有失败状态
- 报告正文最少可展示标题、摘要或 Markdown 正文之一

不合格情形：

- 列表能点开但详情为空白无提示
- 失败与未生成无法区分
- 页面只能显示原始 JSON

## 设置页

完成标准：

- 至少承载与当前版本相关的设置项
- 能承载本地隐私 / AI 配置 / 调试开关中的适用项
- 页面为空时需明确说明“当前暂无可配置项”

不合格情形：

- 预留了页面但完全没有说明
- 暴露明显与当前阶段无关的大量占位项

## 交互级验收标准

1. 主要操作必须有即时反馈，例如保存中、保存成功、保存失败。
2. 关键按钮在不可用时应表现为禁用或无效原因明确。
3. 网络请求中应避免重复提交。
4. 页面返回后，已成功保存的数据应能再次看到。
5. 对用户有后果的操作应尽量给出确认或结果反馈。

## 数据读写验收标准

1. 新增健康事件后，本地数据库中存在对应记录。
2. 新增附件后，详情页可回显附件信息。
3. 提取成功后，详情页可看到结构化结果。
4. 报告生成成功后，报告列表可看到对应条目。
5. 读取失败时，页面显示错误状态，不直接崩溃。

## 本地存储验收标准

1. 附件路径必须可被当前应用访问。
2. 数据库中的附件路径与实际文件状态尽量一致。
3. 删除或失效文件被发现时，页面需有降级提示。
4. 本地保存失败时，用户必须知道失败，而不是误以为已成功。

## AI 输出异常时的验收口径

以下情况均视为已考虑异常处理：

- 请求超时
- 返回非 200 或业务失败
- JSON 非法
- JSON 合法但字段缺失
- JSON 合法但类型错误

最低要求：

1. 页面不崩溃
2. 用户看得到失败提示
3. 当前健康事件保留原始输入
4. 用户可重试或稍后重试
5. 失败状态可被记录

## 错误提示与空状态的最低要求

### 错误提示

- 文案应让用户知道发生了什么
- 文案应包含可执行下一步，如“重试”“返回”“稍后再试”
- 不直接展示原始异常堆栈

### 空状态

- 空状态需说明“为什么空”
- 空状态需尽量提供下一步动作入口
- 不使用纯空白界面代替空状态

## MVP 阶段最低完成标准

满足以下条件才可视为 MVP 子功能完成：

1. 主链路可手动演示
2. 至少有一个成功路径
3. 至少有一个失败路径
4. 本地数据可回显
5. 页面不依赖未完成的云能力
6. 不会因 AI 失败而阻断查看已有记录

## 可暂缓项

以下内容可在当前任务中暂缓，只要不影响主链路可演示：

- 复杂动画
- 高级筛选
- 多语言
- 深色模式细节优化
- 后台自动补算的完整稳定性
- 高级性能优化
- 大范围 UI 统一美化

## 不可暂缓项

以下内容不可暂缓：

- 主链路关键页面能打开
- 本地保存与读取可用
- 关键错误提示存在
- 空状态存在
- AI 异常不崩溃
- 原始输入可回溯
- 文档同步检查已执行

## 文档同步验收

### 必须满足

一个开发任务只有在以下条件都满足时，才算验收通过：

1. 代码改动和受影响文档改动在同一任务中完成。
2. 文档被更新到正确层级：
   - 长期事实写入事实文档
   - 阶段性策略写入 notes 文档
   - 数据口径写入 contracts
   - 完成口径写入 acceptance
3. 最终输出明确列出：
   - 修改了哪些代码文件
   - 修改了哪些文档文件
   - 哪些文档无需更新，以及理由

### 可以不改文档的情况

仅当以下条件同时满足时，才允许不修改任何文档：

1. 本次变更是纯内部重构或纯实现细节修正；
2. 未改变用户可见行为；
3. 未改变数据模型、字段、枚举、JSON、架构边界、通用规范、完成口径；
4. 最终输出明确写出：
   - `No document update required.`
   - 具体理由。

### 不合格情形

以下任一情形都视为验收不通过：

- 改了代码但未执行文档同步检查
- 该改的文档没有改
- 把长期事实写进阶段性文档
- 把临时 workaround 写进长期架构事实
- 最终输出没有说明文档同步结果
