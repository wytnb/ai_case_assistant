# 验收标准

## 文档目的

本文件定义当前版本页面、交互、数据、异常处理和文档同步的最低完成口径。
本文件回答“做到什么程度才算完成”，不描述实现过程和字段细节。

## 通用验收原则

1. 功能必须可演示，不以“代码已写完”作为完成标准。
2. 页面必须覆盖适用的加载、成功、空、失败状态。
3. 本地数据必须可写入、可读取、可回显。
4. AI 相关能力必须在成功与失败两种情况下都可被用户感知和理解。
5. 不要求完美，但不能以崩溃、无提示、无回显、无恢复路径结束流程。

## 当前纳入验收的页面

- 首页
- 健康记录列表页
- 新增记录页
- 健康记录详情页
- 报告列表页
- 报告详情页

以下页面当前不属于验收范围：

- AI 追问页
- 设置页
- 记录编辑页
- 记录删除页

## 页面级验收标准

## 首页

完成标准：

- 能看到应用标题
- 能看到当前版本说明文案
- 能看到三个可点击入口：记录列表、新增记录、报告页
- 不依赖 AI 网络成功才能正常打开

不合格情形：

- 首页没有明确主入口
- 三个主入口缺失任意一个
- 首页打开即崩溃或白屏

## 健康记录列表页

完成标准：

- 能加载并展示本地记录列表
- 为空时展示空状态和“去新增记录”动作
- 加载失败时展示失败提示和重试入口
- 支持从浮动按钮进入新增记录页
- 列表时间按事件结束时间倒序展示
- 成功保存新记录后回到列表页能看到提示反馈

不合格情形：

- 数据为空时页面只有空白
- 列表项无法进入详情
- 加载失败后没有重试路径

## 新增记录页

完成标准：

- `rawText` 必填且支持多行输入
- 支持从相册选择多张图片
- 选图后能看到数量反馈与缩略图预览
- 提交时先执行 AI 提取，再保存到本地数据库
- 若选择了图片，则成功后图片必须完成本地复制和附件入库
- 提交过程中按钮禁用，且用户能看到“正在整理…”状态
- 提交失败时有明确提示
- AI 返回的开始 / 结束时间无效时，必须拦截保存并提示失败

不合格情形：

- 输入后无法判断是否保存成功
- 用户仍需手动填写摘要或备注才能保存
- AI 提取失败时页面直接崩溃
- 选择了图片但保存成功后详情页看不到附件
- 页面离开后没有任何保存结果

## 健康记录详情页

完成标准：

- 能看到事件开始时间
- 能看到事件结束时间
- 能看到记录创建时间
- 能看到来源类型
- 能看到原始文本
- 能看到摘要；若备注缺失则显示“暂无备注”
- 能看到附件列表或“暂无附件”状态
- 图片附件存在时，至少能看到基础图片预览
- 图片加载失败时有明确降级提示
- 记录不存在时有说明文案

不合格情形：

- 详情页无法区分事件时间、原始文本和摘要 / 备注
- 附件加载失败或文件缺失时直接崩溃
- 记录不存在时无任何提示

## 报告列表页

完成标准：

- 能看到报告列表或空状态
- 能切换 `week`、`month`、`quarter`
- 能手动触发当前选中类型的报告生成
- 生成中按钮禁用，并显示进行中状态
- 能打开报告详情页
- 失败时有失败提示

不合格情形：

- 选中报告类型后无法生成对应报告
- 加载失败后无重试入口
- 同一范围反复生成后出现多条重复记录

## 报告详情页

完成标准：

- 能看到标题、报告类型、时间范围、摘要、建议、生成时间
- 能展示 Markdown 原文
- 报告不存在时有说明文案
- `adviceJson` 解析失败时页面不崩溃，可降级为空建议列表

不合格情形：

- 列表能点开但详情空白无提示
- 详情页直接显示原始 JSON
- 报告不存在时没有任何说明

## 交互级验收标准

1. 主要操作必须有即时反馈，例如保存中、保存成功、保存失败。
2. 关键按钮在不可用时应表现为禁用。
3. 网络请求中应避免重复提交。
4. 页面返回后，已成功保存的数据应能再次看到。
5. 对用户有后果的操作应给出结果反馈。

## 数据读写验收标准

1. 新增健康事件后，本地数据库中存在对应记录。
2. 新增附件后，`attachments` 表中存在对应记录，详情页可回显附件信息。
3. 提取成功后，详情页可看到 `eventStartTime`、`eventEndTime`、`symptomSummary`；若 `notes` 缺失，则显示“暂无备注”。
4. 报告生成成功后，报告列表可看到对应条目。
5. 同一报告类型与同一时间范围重复生成时，最终只保留一条有效报告。
6. 读取失败时，页面显示错误状态，不直接崩溃。

## 本地存储验收标准

1. 附件路径必须可被当前应用访问。
2. 图片附件应复制到应用私有目录，不直接依赖选择器临时路径。
3. 文件失效被发现时，页面需有降级提示。
4. 本地保存失败时，用户必须知道失败，而不是误以为已成功。

## AI 输出异常时的验收口径

以下情况均视为已考虑异常处理：

- 提取请求超时或上游失败
- 报告请求超时或上游失败
- 提取接口返回非对象
- 报告接口返回非对象
- 提取接口字段缺失
- 提取接口时间字段无效或顺序错误
- 报告接口字段缺失或类型错误

最低要求：

1. 页面不崩溃
2. 用户看得到失败提示
3. 记录链路失败时不会出现“看似成功但实际上未保存”的状态
4. 已存在的本地数据仍可继续查看
5. 用户可重试或稍后重试

## 错误提示与空状态的最低要求

### 错误提示

- 文案应让用户知道发生了什么
- 文案应包含可执行下一步，例如“重试”“稍后再试”
- 不直接展示原始异常堆栈

### 空状态

- 空状态需说明“为什么空”
- 空状态应尽量提供下一步动作入口
- 不使用纯空白界面代替空状态

## MVP 阶段最低完成标准

满足以下条件才可视为当前 MVP 子功能完成：

1. 主链路可手动演示
2. 至少有一个成功路径
3. 至少有一个失败路径
4. 本地数据可回显
5. 页面不依赖未完成的云能力
6. AI 失败时不会产生误导性的成功反馈

## 可暂缓项

以下内容可在当前任务中暂缓，只要不影响主链路可演示：

- 首页最近记录概览
- Markdown 富渲染
- 记录编辑与删除
- 设置页
- AI 追问页
- 深色模式细节优化
- 高级性能优化
- 大范围 UI 统一美化

## 不可暂缓项

以下内容不可暂缓：

- 主链路关键页面能打开
- 本地保存与读取可用
- 关键错误提示存在
- 空状态存在
- AI 异常不崩溃
- 原始输入可回溯
- 文档同步检查已执行

## 文档同步验收

### 必须满足

一个开发任务只有在以下条件都满足时，才算验收通过：

1. 代码改动和受影响文档改动在同一任务中完成。
2. 文档被更新到正确层级：
   - 长期事实写入事实文档
   - 阶段性策略写入 notes 文档
   - 数据口径写入 contracts
   - 完成口径写入 acceptance
3. 最终输出明确列出：
   - 修改了哪些代码文件
   - 修改了哪些文档文件
   - 哪些文档无需更新，以及理由

### 可以不改文档的情况

仅当以下条件同时满足时，才允许不修改任何文档：

1. 本次变更是纯内部重构或纯实现细节修正；
2. 未改变用户可见行为；
3. 未改变数据模型、字段、枚举、JSON、架构边界、通用规范、完成口径；
4. 最终输出明确写出：
   - `No document update required.`
   - 具体理由。

### 不合格情形

以下任一情形都视为验收不通过：

- 改了代码但未执行文档同步检查
- 该改的文档没有改
- 把长期事实写进阶段性文档
- 把临时 workaround 写进长期结构事实文档
- 最终输出没有说明文档同步结果
